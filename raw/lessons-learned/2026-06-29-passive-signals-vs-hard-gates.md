# Session Reflection: Passive Signals vs Hard Gates

**Date**: 2026-06-29
**Session Goal**: Implement Claude Code hooks for wiki-index lookup, correction detection, and public-repo privacy enforcement.

---

## What Went Well

- The wiki-index-lookup hook was pipe-tested before wiring into settings, catching zero integration bugs at connection time.
- When asked why the previous reflect failed the privacy check, the diagnosis was accurate and immediate: the PreToolUse banner had surfaced the relevant article ("grep for confidential info before writing to a public repo") but the signal was ignored during content authoring.
- The privacy gate pipe-test covered all three cases in one shot: block on sensitive content, pass on clean content, pass on out-of-scope target path.
- The corrections-detection Stop hook correctly handles both flat and nested transcript message formats without requiring a second pass.

## What Went Wrong

- **Passive context was ignored at the critical moment**: The wiki-index-lookup hook injected a directly relevant article into the PreToolUse banner immediately before the offending Write. The content was not checked against that signal. Passive injection is not enforcement.
- **Privacy check treated as a reminder, not a gate**: The session had a known rule (sanitize public wiki content), a fired reminder (the PreToolUse banner), and still violated the rule. The gap between "reminded" and "enforced" was not closed until explicitly asked.

## Lessons Learned

1. **Passive context signals are not enforcement**: A hook that injects `additionalContext` tells the model what to consider — it does not block non-compliant actions. For constraints that must hold (privacy, security, schema validity), a blocking hook with `permissionDecision: "deny"` is the correct tool. Reminders fail under cognitive load; gates do not.

2. **Diagnose why a reminder was ignored before adding more reminders**: When a signal fired and was missed, the answer is usually a gate, not a louder signal. Adding more reminder hooks for the same constraint compounds noise without improving compliance.

3. **Gate scope should match the constraint scope**: The privacy gate was scoped to the public repo directory only — not all writes. This keeps the gate targeted and avoids false positives on legitimate internal writes that happen to contain the same patterns.

4. **Transcript format variation requires defensive parsing**: Session transcript JSONL files contain messages in at least two shapes (flat `{role, content}` and nested `{message: {role, content}}`). A hook that reads transcripts must handle both without assuming one format.

## Action Items

- [ ] When adding a new constraint to a workflow, ask: is this enforced (blocking gate) or advisory (context signal)? Default to a gate for any constraint where violation has already occurred.
- [ ] Before writing content to any public repository path, run a mental scan for: absolute paths, usernames, project-specific identifiers, issue numbers. The gate enforces this automatically now, but the habit should precede the gate.

## Tips & Tricks for Claude Code

- **Tip**: `permissionDecision: "deny"` in a `PreToolUse` hook output blocks the tool call and surfaces `permissionDecisionReason` to the model. Use this for hard constraints, not `additionalContext`, when the constraint must not be bypassed.
- **Tip**: Scope blocking hooks by checking the target file path early — exit 0 immediately if the path is outside the relevant directory. This prevents false positives and keeps hooks fast.
- **Tip**: When a Stop hook needs to read session transcripts, glob for `session_id.jsonl` under the projects directory rather than assuming a fixed path structure.
