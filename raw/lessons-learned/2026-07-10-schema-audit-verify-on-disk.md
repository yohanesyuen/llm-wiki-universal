# Session Reflection: Auditing a vector-DB ingestion schema

**Date**: 2026-07-10
**Session Goal**: Audit an ingestion/DB schema, fix the concrete bugs found, then wipe and re-ingest.

---

## What Went Well

- **Read the whole schema surface before judging.** Read the three modules that share the table (connection/schema helper, MCP server, ingest script) rather than one file, so cross-file issues (a column defined in one place, filtered in another with no index) surfaced instead of just per-file nits.
- **Verified a documented path against the actual filesystem instead of trusting either the code or the docs.** The default env-file path in code did not exist on disk; the project's own doc had the correct path. Catching this needed an `ls`, not a read — a doc/code consistency check that only a live check can settle.
- **Separated clear-cut bugs from judgment calls before acting.** Presented findings ranked by severity, then explicitly asked which to fix now (the two unambiguous bugs) vs. which were scale-dependent improvements — avoided over-fixing on a maintenance task.
- **Proactively flagged a count discrepancy** (log said "indexed N", table had N-11) that the happy-path output hid, tracing it to `ON CONFLICT DO NOTHING` dedup rather than reporting success and moving on.

## What Went Wrong

- **Double-backgrounded a long job.** Launched a slow batch with a trailing `&` *inside* a `run_in_background` tool call. The harness's background tracker only saw the launcher shell return instantly and fired a "completed" notification while the real work (a separate PID) was still running. Had to fall back to `ps -p <pid>` + an `until ! ps` wait-loop to actually track it. One backgrounding mechanism, not two.
- A credential-safety guard blocked an env/profile `grep` mid-audit. It was the right block (the grep could have printed a live connection string), but I only realized the cleaner path — never needing the secret in context at all — after hitting it, not before.

## Lessons Learned

1. **A documented path is a claim, not a fact — `ls` it.** When code has a default file path and a doc has a different one, neither is authoritative until checked on disk. The stale one silently breaks fresh setups (the file simply isn't there), and nothing in either source reveals which is right.
2. **Naive datetime into a `TIMESTAMPTZ` column silently shifts times.** Parsing a UTC timestamp tz-aware and then stripping `tzinfo` before insert makes the DB reinterpret it in the *session's* timezone. Keep it tz-aware end-to-end; the bug is invisible whenever the session tz happens to be UTC, so it hides until it doesn't.
3. **`CREATE TABLE IF NOT EXISTS` is not a migration.** It no-ops against an existing table, so any later column addition written into the create statement never applies. A schema that will ever evolve needs an explicit append-only `ALTER ... ADD COLUMN IF NOT EXISTS` list run alongside the create.
4. **Don't stack two backgrounding mechanisms.** A shell `&` under a harness "run in background" makes the tracker watch the wrong process and report false completion. Pick one.

## Action Items

- [ ] When launching a long job via a run-in-background tool, never add a shell `&`; if I need to poll, wait on the real PID with an `until ! ps -p <pid>` loop.
- [ ] On any schema audit, add "does the default path/file actually exist on disk?" as an explicit check, separate from reading the code.
- [ ] For any `TIMESTAMPTZ` write, grep the producing code for `.replace(tzinfo=None)` / naive `datetime.now()` and treat them as suspects.

## Tips & Tricks for Claude Code

- **Tip**: A secret-materialization guard blocking a `grep` of shell profiles/env is a signal to *remove the secret from the plan*, not to find another command — the audit rarely needs the value in context, only its name.
- **Tip**: When output shows a success count, cross-check it against the actual persisted state (a `COUNT(*)`), not just the log line — `ON CONFLICT DO NOTHING` and similar make "processed" and "stored" diverge silently.

## Generalization Opportunities

- **Skill/checklist**: A reusable "schema/ingestion audit" checklist is starting to emerge — default-path-exists, tz-aware end-to-end, create-vs-migrate, indexes on filtered columns, processed-vs-stored count reconciliation. Worth capturing once the same shape recurs on a third project; premature to formalize from one.
