# Session Reflection: Scripting a Recurring CLI Prompt Pattern

**Date**: 2026-06-29
**Session Goal**: Identify sub-tasks repeated 3+ times via manual commands and propose a script under `~/bin` for them.

---

## What Went Well

- Checked shell history (`~/.zsh_history`) for repeated command patterns rather than relying on memory or assumption.
- Found a genuine repeat: an LLM CLI tool (`qwen-code -p "..."`) invoked with the same prompt template ("Evaluate @<file> — identify bugs, code quality issues, and improvements[. Fix them...]") against different files, 5 times across 2 files.
- Distinguished trivial single commands (`ls`, `cd`, `git status`) from genuine repeated multi-step *patterns* worth automating — only the latter justifies a script.
- Wrote the script as a thin parameterized wrapper (`qwen-eval <file> [--fix]`) rather than hardcoding the observed filenames, since the point of automation is reuse beyond the exact instances observed.

## Lessons Learned

1. **Shell history is the right evidence source for "how many times has X been done."** Don't guess at frequency — grep `.zsh_history` (or equivalent) for the literal recurring command shape before proposing a script.
2. **A repeated pattern only deserves a script if the parts that vary are isolable.** Here, only the filename (and an optional "fix" flag) varied — everything else in the prompt was constant — making it a clean candidate for a one-argument wrapper script.
3. **Single trivial commands repeated many times (`cd`, `ls`) are not "sub-tasks"** — scripting them would just rename muscle memory. The bar for proposing a script should be a multi-token command or prompt template with a constant scaffold and a variable slot.
4. **`~/bin` scripts in this environment follow a consistent minimal style**: `#!/bin/sh`, a one-line comment stating purpose, a `Usage:` comment, default/derived args, then `exec` into the underlying tool so the wrapper doesn't add an extra process layer or swallow exit codes.

## Tips & Tricks for Claude Code

- **Tip**: When asked to find "things done more than N times," grep history files for the *shape* of the command (e.g. `qwen-code -p "Evaluate @`) rather than exact full-string matches, since arguments (filenames) will differ.
- **Tip**: Prefer `exec` at the end of a wrapper script over a bare call, so signals/exit codes propagate and no extra shell process lingers.
