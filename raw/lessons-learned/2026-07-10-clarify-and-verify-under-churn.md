# Session Reflection: Clarifying, Verifying, and Reading Churn Signals

**Date**: 2026-07-10
**Session Goal**: A run of small edits on an experimental module — move a property to a parent class, add type hints package-wide, extend a CLI driver, add a safety backup — interleaved with several low-bandwidth and shifting requests.

---

## What Went Well

- **Structured disambiguation for stable ambiguity.** Several requests used bare deictic references ("shouldn't this go under the parent?", "help me implement this section") with no editor selection attached. Asking a single focused multiple-choice ("which symbol / which section did you mean?") before editing avoided guessing wrong on a consequential change.
- **AST inventory over grep for a code-wide change.** Adding missing type hints started with an AST walk that reported exact per-function param/return gaps. A plain grep for `def` lines lacking `->` produced false positives on multi-line signatures — structural analysis saw actual syntax. Re-running the same AST pass afterward proved zero gaps remained.
- **Verified every change at runtime, not just by eyeballing.** Import checks, a property-resolution check on both parent and subclass, and small unit assertions on a pure helper — each edit was exercised, not assumed.
- **Flagged an unverifiable assumption at the boundary, then closed it.** One annotation needed a third-party type whose import path couldn't be confirmed (the dependency wasn't installed in the interpreter in use). Shipped it behind a `TYPE_CHECKING` guard (zero runtime cost) with the assumption stated explicitly — then confirmed it was correct once the right interpreter (which had the dependency) became available.
- **Adopted a tooling correction immediately and re-verified with it.** When told which interpreter to use going forward, saved it to memory and used it to re-check the earlier assumption rather than leaving it hanging.
- **Refused to fire an outward-facing side-effecting path unprompted.** An experimental driver would drive live, billable actions across real sessions. Exercised only the pure/orchestration parts and left the live run for explicit go-ahead.

## What Went Wrong

- **Offered polished N-way choices for a request the user hadn't actually settled.** On an evolving ask about reshaping an experimental feature, I presented a structured multiple-choice. The user picked an option, then immediately overrode it and interrupted repeatedly as the requirement kept shifting. The structured choice crystallized a decision that was still forming.
- **Missed the interrupt pattern as a signal.** Two-plus interrupts on the same restated request meant "I'm still deciding," not "answer faster." I kept responding in the same structured register instead of dropping to something smaller and more open.

## Lessons Learned

1. **Repeated interrupts mid-refinement mean stop structuring, go smaller.** When a user interrupts two or more times while restating the same request, they're still forming the requirement. Forcing an N-way choice makes them commit prematurely. Better: make the smallest reversible move, or ask one open-ended question, and let them steer.
2. **Structured disambiguation fits stable ambiguity, not unstable intent.** A multiple-choice is ideal when the *referent* is unclear but the *goal* is fixed ("which symbol is 'this'?"). It backfires when the user hasn't decided the goal yet — common for exploratory/experimental code.
3. **Use structural (AST) analysis, not text matching, to inventory code.** Grepping signature lines misclassifies multi-line defs; an AST walk gives exact gaps and doubles as the after-check that the change is complete.
4. **Annotate against an uninstalled dependency behind `TYPE_CHECKING`, then verify the path in an env that has it.** The guard keeps runtime clean and lets the change land; confirming the import path once the dependency is importable closes the open assumption instead of leaving a latent guess.

## Action Items

- [ ] On the second interrupt of the same evolving request, abandon multiple-choice framing — make a minimal reversible change or ask one open question.
- [ ] Default to the project's known-good interpreter for verification from the first check, not the bare system one, so assumptions get confirmed early.

## Tips & Tricks for Claude Code

- **Tip**: To find every function missing a param/return annotation, `ast.walk` the module and check each `FunctionDef`'s `args` annotations and `.returns` — far more reliable than grepping `def` lines, and re-runnable as a completeness check.
- **Tip**: A `if TYPE_CHECKING: from pkg import SomeType` block lets you annotate with a dependency's type that isn't installed in the current interpreter, at zero runtime cost — pair it with `from __future__ import annotations`.
