# Wiki Automation Gaps Found by Comparing Against a Reference Implementation

**Date**: 2026-07-04
**Session Goal**: Compare our LLM-maintained wiki pipeline (raw/ → wiki/, source→compile pattern) against a public reference implementation of the same pattern, to see what's worth adapting.

---

## Context

The reference implementation is a plugin that also follows "LLM curates a wiki, human queries it," but built on top of a note-taking app with PARA (Projects/Areas/Resources/Archive) + Zettelkasten (atomic, backlinked notes) organization. It ships five distinct skills instead of our one, plus a structured programmatic edit surface instead of requiring the compiling skill to run from a single fixed machine.

## What Went Well

- Recognized quickly that the two systems share the same core loop (ingest → compile → query) despite very different substrates, which made an apples-to-apples comparison of *workflow automation* possible without getting distracted by the note-taxonomy differences.
- Correctly scoped what to borrow vs. skip: automation ideas (lint, retopology, decoupled resource-import) generalize regardless of substrate; the PARA/Zettelkasten taxonomy itself does not, since our wiki is narrower in scope (lessons + conventions) than a general personal knowledge vault.

## What Went Wrong

- Nothing significant — this was a comparison/research task, not an implementation task, so no execution mistakes to report.

## Lessons Learned

1. **A single compile skill has no health-check pass.** Our pipeline only ever ingests and grows `wiki/`; nothing periodically checks for broken links, orphaned articles, stale claims, or duplicate topics. The reference implementation's dedicated lint skill catches exactly this class of drift. Worth adding an equivalent lint pass, run periodically or on `/reflect`, that reports (not necessarily auto-fixes) staleness and orphan articles.

2. **Flat topic folders don't get revisited as the corpus grows.** Our topic structure is decided once at ingest time (whatever subdirectory a new article lands in) and never reorganized. A retopology-style pass — re-bucketing or splitting topic folders once one grows past a size where a flat structure stops being useful — is missing entirely.

3. **Ingestion is bottlenecked on a single machine.** The compiling skill only works where it's locally installed, which blocks any other session (e.g. a fresh container, a different repo root) from doing anything beyond dropping a file into `raw/` and leaving a note that ingestion is pending. A more structured, tool-based edit surface (rather than a skill-only compile step) would remove that single point of dependency.

4. **Ingestion and reflection are conflated.** Right now `raw/` sources mostly arrive via the `/reflect` session-retrospective path. There's no separate, first-class path for importing external reading material (articles, other repos, reference docs) that isn't a session reflection — worth keeping those two ingestion styles distinct so neither gets stretched to fit the other.

## Action Items

- [ ] Design a lint pass for `wiki/` (broken links, orphan articles, stale claims, duplicate topics) — report-only to start, matching the deterministic-vs-heuristic split already used by other wiki tooling.
- [ ] Design a retopology pass that reorganizes topic folders once a topic subdirectory grows past a usability threshold.
- [ ] Consider a structured, non-skill-dependent write path for `wiki/` so ingestion isn't bottlenecked on one machine.
- [ ] Keep a distinct ingestion path for external resource imports, separate from `/reflect`-sourced session retrospectives.
