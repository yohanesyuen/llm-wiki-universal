# Session Reflection: Universal Wiki Setup and Rule Import

**Date**: 2026-06-27
**Session Goal**: Bootstrap the universal LLM wiki, wire up the reflect→ingest flow, and selectively import rules from a project-specific wiki into the universal tier.

---

## What Went Well

- **gh CLI for repo exploration** — using `gh api repos/.../contents` to browse a GitHub repo recursively was effective and fast; no WebFetch needed.
- **Numbered ranked list for selection** — presenting candidates as a numbered list with rankings let the user pick by number across multiple turns without confusion or re-listing from scratch each time. (Meta: this was exactly one of the rules we imported.)
- **Incremental import loop** — show list → user picks → ingest → re-list remaining worked cleanly as a multi-turn pattern.
- **Correct universal vs. project-specific triage** — correctly skipped all AutoCAD/COM technical articles and only surfaced the meta/workflow lessons and process conventions.
- **Splitting the reflect skill cleanly** — removing both the marketplace copy and the cache copy of `handbook-extras:reflect` was the right call; the system-reminder immediately confirmed `/reflect` was live.

## What Went Wrong

- **Combined unrelated rules into single articles** — rules 4+7 and 11+12 were bundled into one file each because they came from the same source. User had to explicitly point this out. Violated "one concept per page" from the wiki schema itself.
- **Tried to delete `.obsidian/`** — when asked to "clean up auto-generated Obsidian files," I included the config directory the user wanted to keep. Should have limited cleanup to content files (Welcome.md, create a link.md), not the tool's config.
- **Stale skill name in ~/.claude/CLAUDE.md** — the initial draft referenced `/handbook-extras:reflect` in the reflect→ingest rule, which was already being replaced in the same session. Caught and fixed during the same turn, but required an edit pass.

## Lessons Learned

1. **"One concept per page" applies to ingests too** — when multiple rules come from the same source file, each still gets its own raw file and wiki article. Source co-location is not a reason to merge concepts.
2. **"Clean up X" means content, not config** — when asked to clean up auto-generated files from a tool, default to removing only generated content files. Config directories (`.obsidian/`, `.git/`, etc.) are intentional and should be left alone unless explicitly listed for removal.
3. **Check for self-referential staleness in the same session** — when a session modifies a skill or rule that is itself referenced elsewhere, immediately grep for old references before closing out the work.

## Action Items

- [ ] When ingesting multiple rules from a single source, create N raw files and N wiki articles upfront — don't batch them.
- [ ] When asked to "clean up" a directory, list what will be deleted and confirm before acting if any item is a config or tool directory.

## Tips & Tricks for Claude Code

- **`gh api repos/.../contents`** is the right tool for browsing GitHub repo structure without WebFetch auth issues.
- **Numbered + ranked lists with emojis** make multi-turn selection flows much smoother — the user can say "add 3, 7, 9" without ambiguity, and rankings help them prioritize without having to think from scratch.
