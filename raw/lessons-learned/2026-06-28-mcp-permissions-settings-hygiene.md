# Session Reflection: MCP Permission Setup and Settings Hygiene

**Date**: 2026-06-28
**Session Goal**: Add GitHub MCP tools to the global allow list and clean up accumulated noise in `~/.claude/settings.json`

---

## What Went Well

- **Live-testing before claiming success**: Rather than just saying "the MCP is connected", a real API call was made (`search_repositories`) to confirm authentication and data return. This is the right bar.
- **Validator as teacher**: The first permission edit attempt used wrong syntax (`mcp__github__add_issue_comment(*)`). The settings validator rejected it immediately and printed the exact correct form (`mcp__github__*`). Rather than guessing, the error message was read and applied directly.
- **Two-pass cleanup**: Redundant entries (covered by existing globs) were removed in a first pass, then one-time commands in a second. Separating the two concerns made each pass easier to reason about.
- **Conservative Read entry preservation**: When removing Bash one-timers, all `Read(...)` directory entries were kept — these represent real working directories and are genuinely reusable, not one-time artefacts.

## What Went Wrong

- **Wrong MCP permission syntax on first attempt**: Used `mcp__github__tool_name(*)` (Bash-style) instead of `mcp__github__*` (MCP-style). The rule is: MCP tools don't use parentheses at all; wildcards go directly in the tool name path.

## Lessons Learned

1. **MCP permission rules have different syntax than Bash rules**: Bash rules use `Bash(command *)` with parentheses and interior wildcards. MCP rules use `mcp__server__tool` with no parentheses; `mcp__server__*` covers all tools on a server. Mixing the two patterns silently or noisily fails.

2. **The settings validator gives actionable error messages**: When a settings write is rejected, the error output includes the correct syntax. Read it before trying a different guess — the answer is usually right there.

3. **Allow lists accumulate noise by default**: Every session that hits a new permission prompt adds a new entry. Without periodic review, the list fills with one-time commands (specific find sweeps, `__CMDSUB_OUTPUT__` artefacts, hardcoded paths with specific arguments). A cleanup pass is worth doing whenever the list feels unwieldy.

4. **Two categories of noise in allow lists**:
   - *Redundant*: entries already covered by a broader glob in the same list (e.g., a specific subdirectory path when the parent `/**` is already allowed).
   - *One-time*: exact commands with hardcoded data that were approved once for an investigation and will never recur in that form (find sweeps, introspection prints, test scaffold mkdir/rm).

5. **Prefer glob-shaped entries for recurring tool patterns**: `Bash(pip3 show *)` is more useful than five specific `pip3 show <package>` approvals. When approving a recurring workflow, allow the pattern not the instance.

## Action Items

- [ ] When approving a Bash command at a prompt, consider whether the pattern form would be more durable than the exact command
- [ ] Periodically audit `~/.claude/settings.json` allow list — the two-pass approach (redundant globs first, one-timers second) works well

## Tips & Tricks for Claude Code

- **MCP wildcard syntax**: `"mcp__servername__*"` allows all tools on a server. No parentheses, no glob inside parens.
- **Settings validator gives correct syntax on failure**: When an Edit to settings.json is rejected, the error message includes the right form. Use it directly.
- **Two-pass allow list cleanup**: First remove entries covered by existing globs; then remove one-time exact commands. Doing both at once is harder to reason about.
