# Session Reflection: Statutory Fact Hallucination in Seed Data

**Date**: 2026-07-17
**Session Goal**: Execute Phase 1 of a spec-driven implementation task that included seeding placeholder domain data (labour-law-derived entitlement numbers) into a database.

---

## What Went Well

- When corrected, I re-verified every adjacent claim in the same file rather than fixing only the one number the user flagged — this caught two more wrong/mislabeled figures (one outdated, one mischaracterized) that the user hadn't mentioned.
- Used live web verification (search + fetch of the primary regulator's page) instead of re-asserting from memory once put on notice, and cited sources inline in code comments so future readers/agents don't have to take the claim on faith.
- Wrote a persistent memory capturing not just the corrected fact but the *pattern* (verify statutory/legal claims live, don't recall them) so the same mistake is less likely to recur in a future session on the same or a different project.
- Flagged a second-order data-modeling implication discovered during verification (two related entitlement figures have an "inclusive of" relationship the current schema doesn't represent) as a TODO for later work, rather than silently absorbing it or over-fixing scope in the moment.

## What Went Wrong

- Wrote confident, specific-sounding legal/statutory claims (an exact entitlement number tied to an exact years-of-service threshold) directly into a code comment without verifying them against a primary source first. The specific number was even directionally plausible (in the right ballpark), which made it easy to not double-check.
- This was flagged by the user as a *repeat* pattern across sessions/projects, not a one-off — meaning the underlying habit (treating training-data recall of jurisdiction-specific statutory figures as fact) hadn't been corrected by the first occurrence.
- The error shipped into seed data and a comment before any external verification occurred — verification only happened reactively, after a user correction, rather than proactively before the placeholder was written.

## Lessons Learned

1. **Statutory/legal figures are a distinct risk category from general domain knowledge.** Unlike architectural or coding conventions (where "close enough" placeholders are fine and get iterated), labour-law/regulatory numbers are exact, jurisdiction-specific, and change over time (a cited entitlement had in fact changed within the last ~18 months of the session's simulated timeline). Treat any such figure as a claim requiring citation, not a fact to recall.
2. **A self-critique gate only works if it treats "sounds right" and "is verified" as different states.** The original comment read as confident and specific — exactly the profile that should trigger a live-verification step before being written down, per any standing self-critique/evidence-based-claims policy. The lesson is to apply that policy *before* writing the placeholder, not just before finalizing a response to the user.
3. **One correction is a chance to sweep, not just patch.** Re-checking every adjacent claim in the same file (rather than only the one flagged) surfaced additional errors the user never had to find themselves — this is worth doing by default whenever a factual correction lands on a cluster of related claims (e.g., a table of entitlement figures), since errors in that category rarely occur in isolation.

## Action Items

- [ ] Before writing any code comment, seed value, or spec text that states a jurisdiction-specific statutory/legal number as fact, perform a live source check (web search/fetch against a primary regulator source) in the same turn, and cite the source inline.
- [ ] When a user corrects one factual claim in a cluster of related claims (a table, a list of constants), proactively re-verify the rest of the cluster rather than treating the fix as scoped to the single flagged item.
- [ ] Persist a durable, cross-project memory of this specific failure mode (not just the corrected fact) so it surfaces before similar claims are written in unrelated future sessions.

## Tips & Tricks for Claude Code

- **Tip**: For placeholder/seed data representing regulated or legally-defined values, treat "I recall this number" as a red flag rather than a green light — the more specific and confident-sounding the recalled figure, the more it warrants a live check, since vague/uncertain claims are naturally hedged but precise wrong ones are not.
- **Tip**: When citing a verified fact in a code comment, include the source URL inline — it turns the comment into a checkable claim instead of an assertion, and makes future re-verification (when the law changes again) trivial to locate.
