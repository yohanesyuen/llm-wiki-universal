# Session Reflection: Evaluating MCP-Server Conversion for a Filesystem-Native Knowledge Base

**Date**: 2026-06-28
**Session Goal**: Evaluate pros/cons of wrapping a local, filesystem-native knowledge-base tool (markdown wiki + skill, read/written directly by a coding agent) as an MCP server.

---

## What Went Well

- Verified the actual nature of the target repo (via `gh api`, checking topics and top-level file listing) before evaluating it, rather than relying on the name alone. The repo turned out to be an Agent Skill (filesystem-native, git-backed) rather than a generic web service — that distinction changed the whole analysis.
- Gave a direct recommendation ("not worth it for your current setup") backed by a concrete reason (zero-cost filesystem access already in place) instead of a balanced-sounding but noncommittal list.
- When new information arrived (web client has no filesystem access, so the existing workflow silently fails there), revised the recommendation rather than defending the original take. The second pass kept the same evaluative honesty — ranked options cheapest-to-heaviest instead of just saying "now MCP makes sense."

## What Went Wrong

- The first pass evaluated the conversion against the wrong baseline (single local user, git-native) because the actual trigger — a no-filesystem web client unable to run a local update workflow — hadn't been stated yet. The eval wasn't wrong given what was known, but it missed asking what client surfaces the user actually needs to support.

## Lessons Learned

1. **A capability gap reframes a "is X worth it" question entirely.** "Should we add infrastructure Y" looks like over-engineering until a concrete client/environment is named that literally cannot do the job without it. Ask early whether all relevant client surfaces (e.g., web vs. CLI, with vs. without filesystem access) were considered, since the cost/benefit flips once one surface is excluded.
2. **Reuse existing remote connectors before building new servers.** When the actual gap is "this client can't reach my filesystem," check whether an existing remote integration (e.g., a Git-hosting API/connector) already solves it before scoping a custom server. Building a bespoke MCP server to do what a hosted git API already exposes re-implements a transport layer for no new capability.
3. **Order options by cost before recommending.** When multiple fixes exist for the same gap (use existing connector → build custom remote server → tunnel local machine), present them ordered from cheapest/lowest-maintenance to heaviest, and recommend trying the cheapest first rather than jumping to the most general (and most expensive-to-maintain) solution.

## Action Items

- [ ] Before evaluating whether to add server/protocol infrastructure around an existing local tool, explicitly ask which client environments (filesystem access vs. none) need to use it.
- [ ] When a "convert X to a server" question arises for a git-backed local asset, check first whether the asset already has (or could get) a remote git host with an existing MCP/API connector before designing a bespoke server.

## Tips & Tricks for Claude Code

- **Tip**: Use `gh api repos/<owner>/<repo>` and the contents endpoint to verify a repo's actual shape (skill vs. library vs. service) before reasoning about it — repo names and READMEs can imply a different architecture than what's actually there.
