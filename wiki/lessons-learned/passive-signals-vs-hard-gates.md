# Passive Signals vs Hard Gates

A Claude Code hook that injects `additionalContext` is advisory — the model sees it but can ignore it under cognitive load. A hook that returns `permissionDecision: "deny"` is a gate — the tool call never executes. These are categorically different enforcement mechanisms, and choosing the wrong one for a hard constraint results in the constraint being violated despite the signal firing correctly.

## The Distinction

**Passive signal** (`additionalContext` in `hookSpecificOutput`): surfaces information the model should consider. Fires, injects text into context, and exits. Compliance depends on the model reading and acting on the injected text. Fails silently when attention is elsewhere.

**Hard gate** (`permissionDecision: "deny"`): blocks the tool call entirely. The model receives the `permissionDecisionReason` and must change its approach before the action can proceed. Cannot be accidentally bypassed.

## When to Use Each

Use a passive signal when:
- The guidance is contextual and varies by situation (wiki articles, relevant conventions)
- False positives would be disruptive
- The constraint is advisory rather than mandatory

Use a hard gate when:
- A constraint has already been violated despite a reminder
- The constraint is binary (either safe or not — e.g., privacy, security, schema validity)
- The cost of a violation exceeds the cost of a false positive

## The Diagnostic Rule

If a passive signal fired and the constraint was still violated, the answer is a gate — not a louder or more prominent signal. Adding more advisory hooks for the same constraint compounds noise without improving compliance.

## Scoping Gates

Gate scope should match constraint scope precisely. A privacy gate on public repository writes should exit immediately for any target path outside that directory — otherwise it triggers false positives on legitimate internal writes that happen to contain similar patterns. Narrow scope keeps gates fast and avoids friction on unrelated operations.

## False Positive Risk in Gate Patterns

Regex patterns in blocking gates must be tight. Overly broad patterns (e.g., matching any sequence of hyphenated words) will block legitimate content — wiki filenames, article slugs, cross-references — producing false positives that erode trust in the gate. Test every gate pattern against representative clean content before wiring it into settings.

## A Denial Isn't Necessarily a Permanent Wall

`permissionDecision: "deny"` blocks the specific evaluated call every time it's evaluated as-is — that part of the hard-gate framing holds. But a safety/permission classifier's decision can incorporate recent conversational context, not just the isolated command: the same low-level action denied once has been observed to succeed on retry after the user pushed back with explicit, forceful re-confirmation, with no code or pattern change in between. Treat a denial as final for the *current* context — stop, explain the concern plainly, and let the human decide — but a genuinely re-confirmed retry is a new evaluation, not a bypass of the old one, and is reasonable to attempt.

## Tips & Tricks

- **Tip**: `permissionDecision: "deny"` with a `permissionDecisionReason` is the correct output shape for a blocking `PreToolUse` hook. The reason is surfaced to the model, not just logged.
- **Tip**: Exit 0 with no output for out-of-scope inputs — the hook framework treats silent exit 0 as "allow."
- **Tip**: Always pipe-test the gate with three cases: should-block, should-pass, and out-of-scope. The third case catches over-broad matching.
