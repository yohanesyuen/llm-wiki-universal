# Allowlist Audit and Session Hygiene

Lessons from auditing accumulated Claude Code patterns to reduce friction — removing one-time allowlist entries, adding a /reflect stop-hook reminder, and scanning transcripts to find genuinely recurring permission prompts worth pre-approving.

## What Went Well

- **Evidence-first pattern audit**: Rather than guessing at what commands recur, the session parsed `~/.claude/history.jsonl`, recent session `.jsonl` transcripts, and lessons-learned files together. This produced concrete counts instead of hunches.
- **Dual-source transcript analysis**: Both shell history and session transcripts were scanned, catching Claude-driven tool calls that never appear in shell history.
- **Minimal targeted changes**: When specific items were requested, only those were done — no extra scope was added.
- **Permission scan revealed a non-obvious gap**: `git -C /path subcommand` is NOT caught by Claude Code's built-in git auto-allow. The read-only validator checks the subcommand at position 2, but `-C /path` pushes the subcommand to position 4. This appeared many times across transcripts and was the largest friction source found.
- **Filter discipline**: The scan correctly excluded mutation and interpreter commands even though they appeared frequently — preventing over-permissive allowlist entries.

## What Went Wrong

- **First-pass frequency script was too aggressive in filtering**: The initial extraction script dropped MCP tools whose names didn't contain read/get/list/search/view/fetch, silently excluding high-frequency tools. A second pass surfaced them for deliberate exclusion. The final decisions were correct, but the first pass should have flagged them for review rather than silently skipping them.

## Lessons Learned

1. **`git -C path subcommand` is not auto-allowed**: Claude Code's read-only validator for `git` expects the subcommand at position 2. When `-C /path` precedes the subcommand, it lands at position 4 and the validator falls through to a permission prompt. The fix is explicit patterns: `Bash(git -C * status)`, `Bash(git -C * diff *)`, `Bash(git -C * log *)`, etc.

2. **Allowlist accumulates noise by default**: Every session that hits a new permission prompt can add a new entry. Without periodic review, one-time commands pile up alongside genuinely reusable glob patterns. Two-pass cleanup — first remove entries already covered by broader globs, then remove one-timers — is a reliable method.

3. **Stop hooks are the right mechanism for session-end reminders**: A reminder buried in a memory file or CLAUDE.md is ignored at session end. A `Stop` hook fires every time the agent returns control to the user — it is the correct place for "did you remember to reflect?" prompts.

4. **Threshold for allowlist entries should be ~3 uses**: Patterns appearing once or twice don't justify an entry — the prompt is infrequent enough that it's not a real friction point. The 3+ use bar keeps the allowlist from growing back into noise.

## Tips & Tricks for Claude Code

- **Tip**: To audit your own recurring patterns, parse `~/.claude/projects/**/*.jsonl` for `type: "tool_use"` entries in assistant messages — this catches everything Claude does, not just what the user types into the shell.
- **Tip**: When a `Bash(git ...)` command keeps prompting despite the subcommand looking safe, check whether flags like `-C` or `--work-tree` are shifting the subcommand out of position 2.
- **Tip**: `Stop` hooks in `settings.json` are session-end triggers. Use them for reminders or lightweight automation that should run after every Claude turn.
